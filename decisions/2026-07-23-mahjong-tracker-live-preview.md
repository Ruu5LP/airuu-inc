# 2026-07-23 麻雀トラッカーに「入力中プレビュー（実況）」を足した

## 決めたこと

[[麻雀トラッカー]]（[Ruu5LP/MaaNanoJa](https://github.com/Ruu5LP/MaaNanoJa)）の [[麻雀トラッカーを同じWiFi内でPC・スマホ同期できるようにした|LAN同期]] に続けて、
**「誰かが入力している最中の半荘」を、他の全端末の画面上部にライブで映す**機能を足した。main にマージ済み（PR #3）。
Ruuの「画面が同期して欲しい」という要望から。

## 背景 / 選択肢

LAN同期でデータ（確定した半荘）は揃うようになったが、**半荘は「終了→保存」するまで DB に入らない**。
そのため「今まさに入力している最中の半荘」は他端末からは見えなかった。Ruuが欲しかったのはそこ＝入力の途中を大画面で共有すること。

「画面が同期」には解釈が複数あったので、作る前に**使う場面を2回に分けて確認**した（これが効いた。下の学び参照）:

1. 何を同期？ → 「タブ表示」ではなく **「入力の途中を大画面に映す」**
2. PC側の見え方は？ → 専用画面ではなく **「どのタブでも入力中にポップで出る」**

## やったこと

- 確定データ(DB)とは**別の「実況」チャンネル**（`/api/live`）を追加。入力中の端末が「進行中の半荘（スコアボード＋打っている最中の1局）」を流し、他端末が受けて画面上部にライブ表示する。
- 実況はサーバの**メモリ上だけ**に持ち、`db.json`・ファイル・外部には一切残さない（消えていい一時状態＝プレゼンス）。落ちても記録には影響しない best-effort。
- 鮮度判定はサーバ時計（応答の `ts`/`now`）で行い、端末間の時計ズレを回避。判定は `lib/live.ts` の純粋関数に切り出してテスト。
- 入力フロー本体は変えず、外向きの「今の入力内容を知らせる」報告だけ足した。確定・半荘終了・画面離脱で実況は自動的に消える。

## 学び

- **「同期して欲しい」みたいな曖昧な要望は、"使う場面"に分解して2段で聞くと外さない。** 今回は「何を」「PCではどう見える」を分けて確認したことで、実装は一本道になった。要望の言葉そのものではなく、その裏の"卓を囲んでいる場面"を一緒に具体化するのが近道（[[UI/UX設計原則]] の「実際の使用シーンをシミュレートする」を、作る前の確認に効かせた）。
- **性質の違うデータは、同期の仕組みごと分ける。** 確定データ(DB)は「消えたら困る・last-write-wins」、実況は「消えていい・プレゼンス」。これを同じ箱に入れると、DB同期を実況のノイズで汚す。**箱を分けたことで、片方の割り切りがもう片方を壊さない**構造になった。前回の「LAN完結」の芯を保ったまま拡張できたのも、この分離のおかげ。
- **時刻の"正"を1つに決める。** 鮮度を各端末の時計で測ると、端末間のズレで「まだ入力中なのに消える／消えたのに残る」が起きる。サーバ時計を唯一の基準にするだけで、その手のバグを構造で消せた。分散の小さな話でも「時計は1つ」。
- **役割分担が効いた実感**: Ruuが「あまりわかってなくてごめん」と言っていた領域でも、**場面さえ一緒に具体化できれば、判断はRuuができる**。技術の中身はこちらが握り、決めるのはRuu、という [[創業者とCEOの関係協定]] の型がそのまま機能した。
- **「早く触ってもらう」が一発で効いた実例（後日追記）**: 初版は「入力中は画面上部に小さくポップ（バナー）」で出した。Ruuが実機（`npm run host`）で触ってすぐ「思ってたのと違う、下半分が変わらない」と指摘。バナーだけだとメイン領域（新しい半荘フォーム）が入力に連れて動かず、"画面が一緒に動いてる"感が出なかった。→ **記録タブの下半分そのものを「観戦画面」に置き換え**（大きなスコアボード＋これまでの局ログ＋今打っている1局をフル表示）て解決。事前の質問（何を／どう見える）だけでは詰めきれず、**実機を触った瞬間に正解が分かった**。[[麻雀トラッカーの局ログ入力をRuuの指摘で作り直した|動くこと≠使いやすいこと]] の教訓どおり、UIは"完成度"より"触ってもらう速さ"。実機で触るたびに現実とのズレが1つずつ出て、そのたびに小さく詰めた——「バナー→観戦画面」「対局中だけ→席選び段階から実況」「再ビルドしても古い画面が出る→エントリHTMLをキャッシュ無効に」。どれも**作り直しではなく既存の仕組みの流用・追加**で済み、1回あたりの手戻りは小さかった（バナーの中身をそのまま観戦画面に、実況ペイロードに段階(phase)を足すだけ、等）。**"完璧に作ってから見せる"より"荒くても早く触ってもらって1個ずつ直す"方が、トータルで速く・的を外さない**という実演になった。
- **他人の状態を"映す"機能が、その人自身の"操作面"を奪ってはいけない（強い原則）**: 席選び段階の実況を足したら、記録タブが「別端末が入力中」なだけで画面まるごと観戦ビューに差し替わり、Ruuの入力フォームが消えて真っ白な「入力を待っています…」に占領された。原因は"観戦＝本体を置き換える"という設計。→ **観戦は自分の入力フォームの"上に添える"だけ**に変え、フォームは常に残す（入力面は絶対に消さない）。さらに"中身のある実況のときだけ"出すよう純粋関数で絞った（空/壊れた実況で他端末を占領しない）。教訓を一般化すると: **プレゼンス（誰かの今）を他者の画面に出すときは、受け手の主導権を奪わない＝"足す"はしても"消す・乗っ取る"はしない**。ここも実機で触った瞬間に「これは無い」と即分かった類のズレだった。
- **同じアプリでも"端末ごとに最適な見せ方"が違う（レスポンシブは機能）**: 「PCの大画面で四分割・一画面に見たい」という要望。スマホの縦1カラムをそのままPCに映すと余白だらけで一望できない。→ **PC幅（≥1024px）のときだけ2×2**にした（成績タブ＝4カードを2×2／記録タブの観戦中＝観戦盤＋局ログ｜今節の成績、の2カラム）。肝は**スマホ側は一切変えない**こと——入力の主戦場はスマホなので、そこは慣れた1カラムのまま、PCは"みんなで見るモニター"として大画面を活かす。1つのコードで役割の違う2画面を両立させるのに、`@media` と「既定は狭く中央寄せ／モニター系だけ全幅」の切り分けが効いた。あわせて**着順分布は色帯だけに頼らず数値（1位 n・xx%）を併記**して判読性を上げた（色覚多様性＋帯が細いと読めない問題の保険）。レイアウトはテストで崩れを拾えないので、実機幅（1360px/390px）で描画して目視確認してから完成にした。
- **共有の"1枠"プレゼンスは、書き手が自分で読むと点滅する（単一スロットの取り合い）**: 実況（誰が今入力中か）はサーバの1枠 last-write-wins。家麻雀で2端末が同時に発信（PCが席選び中＋スマホが局ログ入力中）すると枠が交互に上書きされ、受信側で観戦画面が出たり消えたりで**点滅**した（実機でRuuが指摘）。→ **自分が発信中の端末は他端末の実況を出さない**（発信中は入力に集中＝観戦不要）。観戦専用の端末だけが安定して1人ぶんを映す。教訓の一般化: **共有の単一スロットなプレゼンスは、"書いている人"には見せない（見せると自分の書き込みと他人の書き込みの入れ替わりで点滅する）。表示するのは純粋な観測者だけ**。関連で、状態を持つUIは表示条件が変わっても同じ`key`で描かないと再マウントで入力が消える（席選びが飛んだ）ことも実機で確認。この点滅も「実機で触った瞬間に分かる」タイプで、テストは緑のままだった。

## 次の一歩

- 本番（2026-07-25の家麻雀）で実際に使って、プレビューの出方・畳み方の使い心地を確認。手戻りは早いほど安い。
- もし「複数人が本当に同時に別々の半荘を入力する」運用になったら、実況もDBも last-write-wins なので後勝ちになる。そのときは設計から見直す（[[麻雀トラッカーを同じWiFi内でPC・スマホ同期できるようにした]] と同じ割り切りライン）。
