# 2026-07-23 麻雀トラッカーを同じWiFi内でPC・スマホ同期できるようにした

## 決めたこと

[[麻雀トラッカー]]（[Ruu5LP/MaaNanoJa](https://github.com/Ruu5LP/MaaNanoJa)）に、**同じWiFiにつないだPC・スマホから同じデータを見て・入力できるLAN同期モード**を足した。
明後日（2026-07-25）の家麻雀で、**PCのモニターに全員ぶんを映しつつ、入力はスマホでもPCでもできる**ようにしたい、というRuuの要望から。main にマージ済み（PR #2）。

## 背景 / 選択肢

初版はバックエンドなし・データは各端末の localStorage。だから端末をまたぐと中身が別々で、連動しない。
「連動させたい」に対して選択肢を3つ出した:

1. **PCを親機にしてLAN内サーバを立てる**（PCがデータを持ち、スマホはPCのIPを開く）
2. P2P同期（WebRTC）… サーバ不要だが接続が不安定で、月イチ家麻雀にはオーバースペック
3. 現状のまま手動でJSON書き出し/読み込み … ゼロコストだが「連動」ではない

Ruuが「**PCモニターに映して、入力はどっちからでも**」と使う場面を具体的にくれたので、**①に即決**。
「バックエンドなし」の看板は下ろすことになるが、**外部クラウドには一切出さず"LAN内だけで完結"**するなら、プライバシーの芯は保てると判断した。

## やったこと

- `server/server.mjs` … **依存ゼロ**（Node標準のみ）のLAN同期サーバ。`dist/` を配信し `GET/PUT /api/db` で共有DBを出し入れ。「依存は最小限」を守るためExpress等は入れない。
- `src/lib/remote.ts` … サーバ通信の境界。取り込み判定は純粋関数に切り出してテスト付き。
- `src/useLanSync.ts` … サーバが居れば `db` を同期するReactフック。起動時に `/api/db` を叩いて**サーバの有無を自動判定**し、居なければ従来どおり localStorage だけで動く（＝普段の挙動は不変）。
- ヘッダに同期状態バッジ（📶 LAN同期中 / 📴 この端末だけ）を追加。
- 当日は **`npm run host` 一発**でビルド＋起動。ログに各端末で開くURLが出る。
- 割り切り: 同期は **DB全体＋版数の last-write-wins**。完全に同時刻の別入力は後勝ち。家麻雀は「1局を誰か1人が入力する」流れなので実質起きない前提。限界は `CLAUDE.md` に明記した。

## 学び

- **設計がきれいだと拡張が安い。** データが `db` 1個・保存が `saveDB` 1箇所という"choke point"だったから、UIロジックをほぼ触らず同期を差し込めた。MaaNanoJa の受け入れ基準「次に触るAI（や人）が理解しやすく・壊しにくい状態か」は、そのまま**後からの機能追加のしやすさ**になって返ってきた。読みやすさは利子がつく。
- **既存の設計思想と衝突する追加は、勝手に踏み越えず場面を聞く。** 「バックエンドなし」に反する話だったが、無数の実装案を並べる前に「どういう場面で連動させたい？」を先に聞いたら、選択肢を1本に絞れた。**要望の解像度は"使うシーン"に宿る**（[[UI/UX設計原則]] の「実際の使用シーンをシミュレートする」を、作る前の設計選択にも効かせた形）。
- **制約は言葉の解像度を上げると両立できることがある。** 「ローカル完結」を「**LAN完結**」に読み替えたことで、プライバシーの芯（外に出さない）を保ったまま要望を満たせた。守るべきは"言葉"ではなく"その裏の意図"。
- **割り切りは隠さず書いて次の人に渡す。** last-write-wins の限界を `CLAUDE.md` に明記した。動く≠終わり。どこで壊れるかを書いておくのが、[[麻雀トラッカーの局ログ入力をRuuの指摘で作り直した]] の教訓の続き。

## 次の一歩

- 本番前に一度 `npm run host` でリハーサル（スマホで開いて触り心地確認）。手戻りは早いほど安い。
- もし将来「複数人が本当に同時入力する」運用になったら、フィールド単位のマージ（CRDT等）が要る。そのときは設計から見直す。
- 家の外・別WiFiからの同期（クラウド）は今回やっていない。欲しくなったら別途検討。進行は [[麻雀トラッカー]] を参照。
