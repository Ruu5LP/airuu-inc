# ループエンジニアリング

「AIに毎回プロンプトを書く」のをやめて、「AIを繰り返し回す**ループそのもの**を設計する」側に立つ、という2026年6月に名前がついた開発の考え方。Ruu が興味を持ったので、まず正典として要点を残し、このリポジトリで回せる**安全な最小ループ**を1個設計しておく。

## 目的

- ループエンジニアリングの本質（と限界）を、後から参照できる形で1ノートに固める。
- 抽象論で終わらせず、airuu-inc の中で「検証可能・可逆・コスト上限つき」の最小ループを**設計**し、いつでも回せる状態にしておく。

## 状態

**学習中 / 設計フェーズ。** 概念の整理と最小ループの設計まで完了。実際に自動で回す（オートメーション化）のは Ruu の GO 待ち。

## ループエンジニアリングとは（要点）

火付け役の言葉が一番わかりやすい。

- **Peter Steinberger**：「もうコーディングエージェントにプロンプトを書くな。エージェントにプロンプトを出す"ループ"を設計しろ」
- **Boris Cherny**（Anthropic / Claude Code責任者）：「僕はもうClaudeに直接プロンプトしてない。ループを走らせてて、僕がプロンプトしてるのはそのループのほうだ」
- 体系化したのは元Googleの **Addy Osmani** のエッセイ（6/7、O'Reilly Radar が 6/22 再掲）。

背景：2026年前半で「プロンプトの言い回し」がボトルネックじゃなくなった。エージェントがツールを呼び何十ステップも自律で動くようになると、壊す原因は「言葉のマズさ」ではなく、**止まらない・記憶がない・自己チェックがない**ループ設計そのものになった。プロンプト → コンテキスト → ハーネス → **ループ** という流れの、いま最前線。

### ループを構成する 5＋1 の部品（Addy 版）

Addy はこれを Claude Code の実機能に対応させて説明している。**全部この環境に実在する**のがポイント。

| 部品 | 役割 |
| --- | --- |
| オートメーション | 定期実行＝「ループ」にする心臓 |
| ワークツリー | 隔離された並列作業コピー |
| スキル | プロジェクト知識を `SKILL.md` に固める |
| プラグイン／コネクタ | 実ツールへの接続（MCP） |
| サブエージェント | 独立した検証役 |
| ＋外部状態 | markdown やボードで、周回をまたいで進捗を残す |

最後の「外部状態」が地味に一番大事。ループはメモリを失うと毎回ゼロからになるので、進捗をファイルに落として次の周回が読む設計にする。**airuu-inc の記録運用（[[決定ログ]] / [[振り返り]]）が、そのまま loop の外部メモリになる。**

### 検証役（ベリファイア）とストップルール

- ループは「エージェントが終わったと思ったから」止まるのではなく、**「検証役が合格を出したから」止まる。**
- **自己採点は数えない。** モデルは自分の仕事の判定が下手。*別コンテキスト*のサブエージェントにルーブリックで採点させる。
- 止め方は3種類：**成功**（ゴール達成）／**降参**（例: 5回試してダメならやめる）／**ハンドオフ**（危険・曖昧なら推測せず人間へ）。

## このリポジトリでの「安全な最小ループ」設計（教材）

題材：**ノート整合性ループ**。airuu-inc が自分で決めた規約（[[CLAUDE.md|記述規約]]）を、リポジトリ自身に守らせるループ。検証が機械判定でき、可逆で、コスト上限を引きやすい——最小ループの教材としてちょうどいい。

**検査する不変条件（＝ゴール）**

1. すべての `[[wikilink]]` が、実在するノートか alias に解決する（協定の「既存の wikilink を壊さない」）。
2. 各ノートに frontmatter（`title` / `tags` / `aliases`）がある。
3. ファイル名・フォルダ名が ASCII。

**5＋1 の部品を埋めると**

- **トリガー**：週次（例: 日曜夜）か、main へのマージ時。
- **トポロジー**：単体エージェント＋検証サブエージェント1つ。並列は不要な最小構成。
- **スキル**：検査基準＝ [[CLAUDE.md|記述規約]]。毎回説明せず基準として渡す。
- **プラグイン／コネクタ**：GitHub 連携（ブランチ・PR）。
- **サブエージェント（検証役）**：上の1〜3を別コンテキストで採点。全部パスで合格。
- **＋外部状態**：検査結果を振り返りノートに追記。周回ごとの差分（新しく壊れたリンク等）が見える。

**ストップルール**

- **成功**：3チェックすべて緑。
- **降参**：同じ壊れを3回直せなければ、人間（Ruu）へ。
- **ハンドオフ**：リンク先の"正解"が曖昧なとき（どのノートに繋ぐべきか機械で決められない）は、勝手に消さず Ruu に確認。

**安全レール（ここを外さない）**

- **可逆**：触るのは markdown だけ・すべて git 管理下・**ブランチ／PR 経由**で直接 main を触らない。
- **コスト上限**：1周の最大反復5回・対象ファイル数の上限を決める（暴走＝壊れたツールを連打し続ける事故を防ぐ）。
- **不可逆操作なし**：ノート削除やリンク一括置換は自動でやらず**提案止まり**。承認の輪に人間を残す。

この設計なら、流行の「自走ループ」を体で理解しつつ、**間違えても markdown が少し直るだけで済む**。危ない要素（送金・削除・外部公開）を一切含まないのが選定理由。

## 気をつけること（懐疑）

流行に乗る前に、実際に回している人たちの正直な指摘：

- **直線で済むならループにするな。** 直列で終わる仕事をループで包むと、コスト・レイテンシ・故障モードが増えるだけ。
- **検証できないゴールはスロットマシン。** 主観的・人間しか判定できないゴールだと、適当に止まるか永遠に回る。
- **コストが跳ねる。** 単体で通常チャットの約4倍、マルチエージェントで約15倍。実例：壊れたツールを5分で400回叩き続けたケース。→ コスト上限は必須。
- **サイレント故障。** 自信満々の出力を出しながら1ミリも前進していないことがある。外形の検証が要る。
- **不可逆な操作に回すな。** メール送信・送金・削除は人間を承認の輪に残す。

本番の失敗の大半は4つに集約される：**①ハードな停止条件がない ②ゴールの詰めが甘い ③長時間でコンテキストが溢れる ④コスト管理がない。**

## 次の一歩

- 上の「ノート整合性ループ」を、**まず手動で1周**回してみる（自動化の前に、検証役の採点がちゃんと機能するか確認）。
- 手応えがあれば、週次オートメーション化を Ruu が GO するか決める。まず直線で足りないか疑うのを忘れずに。
